Implementation with the Rover
The idea: Have the rover look around, detect whenever the tag is in view, and when it is, have it adjust itself accordingly.
Roll is going to be the most important axis of rotation for us. Roll is how much the code is rotated left or right when its facing away from a wall. Roll in theory will be able to tell us what angle we are seeing it at, once we put it in the center of the screen.

The idea:
Have a "scanning" mode. Have the rover either spin around or move a little to find a tag. Then, once a tag is found, use offset info to center the code in respect to the camera. Next, use roll information to have it look at the code facing straight ahead. Finally use distance info to get to a specific point away from the code.
Note: Roll may or may not be needed. It is most needed if we need to look at the code from a specific angle.

IMPORTANT THING TO NOTE:
You will have to change the OpenCV Input for all code. I have 2 ideas as to how to do this.
1. Make all the code work through the ESP32 and rover instead of communicating to a laptop. This MIGHT work, but will also most likely run much laggier. For Aruco codes, we want relatively good latency and video footage, so this may not be optimal. It will be easier to work with for OpenCV however, and less annoying message sends.
2 (My suggestion). Have OpenCV use the web server, then send information about where to go through socket, to the ESP32. Slightly more complicated in the short run, however, makes the most sense.


How to program the robot:
Have an initial, "scanning phase"
Once a code is found, you could even program it to a specific ID, then have it rotate either left or right depending on the offset, until it is centered.
Have the robot move towards the code until it is a specific distance away, then repeat with a scanning mode again
Have the code mark down the completed IDs. If an ID has already been went to, have the robot skip over it.
